Key Concept: Labels
[diagram]
Labels are a key/value pair that allows for loose coupling of related objects together.
Standard Use-Case:
  • To group together related objects, assign them one or more matching labels. app=api
More Advanced:
  • Broad label queries allow you to easily run mixed versions of your app.
  • Specific label queries allow you to quickly cut traffic over from one version to another.
Key Concept: Pods
[diagram]
A pod is a collection of containers that run together as a unit. A pod shares networking and host resources among itself.
Standard Use-Case:
  • Run a single container that is accessible on a certain port
More Advanced:
  • Containers within your pod can communicate with each other. For example, your app server can talk to a cache running in the same pod.
  • Each pod has its own IP address. This means you can run software on standard ports without conflict.
Key Concept: Services
[diagram]
Services automatically load-balance traffic between a set of pods, specified by a label query.
The label query for a service can be changed at any time without consumers of the service noticing.
Standard Use-Case:
  • To expose a service to the cluster, you define a label query that matches backends based off that query.
More Advanced:
  • Unlike DNS, service membership is real-time and always reflects the qualified pods.
  • Services also have labels, making it easy to programatically manipulate them.
Key Concept: Replication Controller
[diagram]
Many copies of a pod can be run with a Replication Controller.
Pods will be created or destroyed to maintain the correct number of replicas. A label query is used to count the pods.
Standard Use-Case:
  • Define a pod template and set the active count to 3.
More Advanced:
  • When releasing a new version, create a new replica controller and slowly increment the replica count as you verify that everything is working properly.
